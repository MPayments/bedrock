# Текущая архитектура

_Состояние на 2026-03-01._

## Коротко

Bedrock - это framework для детерминированной финансовой обработки:

- API принимает команды и валидирует права
- Документы фиксируют workflow-состояние
- Connectors исполняют внешние платежные действия
- Orchestration выбирает оптимального провайдера
- Ledger/Accounting/Balances держат финансовую консистентность

## Главные архитектурные принципы

## 1. Closure factories вместо классов

Все сервисы и handlers создаются как функции-фабрики (`createXxxService`).

## 2. Документы - источник workflow-правды

Workflow-состояния и переходы живут в document-модели, а не в разбросанных флагах.

## 3. Платежный runtime явный

Платежный контур построен вокруг:

- `@bedrock/connectors`
- `@bedrock/orchestration`
- `@bedrock/payments`

## 4. Асинхронная сходимость

Запрос в API фиксирует состояние и намерение, а внешние сайд-эффекты доводятся воркерами.

## Плоскости системы

- **API/Auth**: `apps/api`
- **Workflow**: `@bedrock/documents` + document modules
- **Connectors**: провайдеры, попытки, webhook, polling
- **Orchestration**: правила маршрутизации и fallback
- **Financial execution**: accounting + ledger + balances
- **Reconciliation**: сверка внешних движений
- **Module runtime**: включение/выключение компонентов

## Что это дает на практике

- предсказуемые переходы состояний
- простые audit trails
- контролируемые ретраи
- прозрачную модульность по компонентам
