# Жизненный цикл платежа

Ниже базовый сценарий `payment_intent -> payment_resolution`.

## Шаги

1. Клиент создает черновик платежа.
2. Выполняет workflow-переходы (`submit`, `approve`).
3. На `post` система:
   - делает финансовый pending path
   - создает connector intent
   - запрашивает orchestration route
   - ставит первую attempt в очередь
4. Воркеры dispatch/polling/webhook двигают попытки.
5. На терминальном результате создается `payment_resolution`.

## Пример: создание платежа

```bash
curl -X POST http://localhost:3002/v1/payments \
  -H 'Content-Type: application/json' \
  -H 'Cookie: <session-cookie>' \
  -d '{
    "direction": "payout",
    "amountMinor": "150000",
    "currency": "USD",
    "corridor": "US-RU"
  }'
```

## Пример: workflow-переход с идемпотентностью

```bash
curl -X POST http://localhost:3002/v1/payments/<paymentId>/post \
  -H 'Content-Type: application/json' \
  -H 'Cookie: <session-cookie>' \
  -H 'Idempotency-Key: payment-post-<paymentId>-v1'
```

## Как читать статус

- `intent` показывает исходное бизнес-намерение
- `attempts` показывают историю отправок провайдерам
- `events` показывают webhook/polling изменения
- `resolution` показывает итог (`settle`, `void`, `fail`)

## Почему это удобно

- любой шаг можно повторять безопасно (через idempotency key)
- причины терминального статуса видны в событиях
- отладка проще: документ + попытки + webhook trail
